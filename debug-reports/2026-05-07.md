**What went wrong**

There are really only two root causes here; everything else downstream is a consequence of these two compounding.

1. **Planner over-disambiguated when it had the answer.** RAMONA is the Burger Master participant. The catalog explicitly carries this — RAMONA's tags/description state it. So when the customer wrote "hamburguesa del burger master", the planner had enough signal to emit `ADD_TO_CART(RAMONA)` directly. Instead it treated the phrase as ambiguous and asked the customer to confirm. The Coke was added in the same turn but the burger was deferred to a follow-up that never happened (because the customer was busy sending the next 5 messages — see #2). **Fix shape**: when SEARCH_PRODUCTS returns exactly one product with a strong tag-match for the qualifier (`burger master` → exactly 1 product tagged `burger_master`), the planner should ADD_TO_CART instead of asking. The disambiguation prompt should only fire when the candidates are genuinely tied.

2. **Debouncer window is fixed-short and doesn't react to typing bursts.** The current debouncer has the right primitives — abort + coalesce + a short wait window. But the window doesn't widen when the user is clearly mid-burst, so a turn ran while the customer was still typing the disambiguation answer. The planner started racing the human, and from there everything went sideways: order placed without burger, customer protested, bot re-listed burgers as if the prior context didn't exist. **Fix shape**: when a new message arrives during the wait window, extend the window (exponential backoff, capped at e.g. 8s). When two consecutive messages arrive within sub-second cadence, treat it as a typing storm and wait for a real pause (e.g. 3s of silence) before running the planner. This could be deterministic (timing-only) or planner-assisted (LLM gets to vote "the user looks mid-thought, give them another 2s"), but the deterministic version is enough for 95% of cases.

The other things that happened in this conversation are downstream:
- Phone stored as `324273929292` (duplicated digits): no validation on the phone field, but it didn't cause harm here. Worth fixing alongside the +573155911909 citizen-ID-as-phone case.
- Bot's reply "Tu pedido quedó registrado y está pendiente de confirmación. El total es de $12.500 ..." doubling down on the wrong total: this is the bot reading the cart back honestly. The cart was wrong because of (1) and (2). Don't blame the response generator for being faithful.
- Operator pasting the bot template at 19:25:10: same template-paste anti-pattern we've seen all week, and same fix surface (`483b36b` admin "Crear pedido" + reactivate/amend).
- Post-delivery $7k over-charge + missing Coke + Nequi refund: this is operations, not a bot bug. Bot's reading of the complaint was bad ("se cobra aparte" when the customer said the Coke wasn't delivered) — that's the post-confirmation cart-mutation issue we've already flagged elsewhere.

The headline is: **the planner needs to trust strong tag matches, and the debouncer needs to be typing-storm aware.** With those two in place, this entire 18-message disaster collapses into a normal happy-path conversation.
